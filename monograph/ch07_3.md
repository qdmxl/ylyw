# 7.3 "卦象意图"通讯协议

第7.2节定义了三个层级的YLYW元胞各自独立运行、共享相同的推理架构。但层级之间如何通讯？宏观层"告诉"中观层什么？中观层又"告诉"微观层什么？如果每层是独立运行的完整推理引擎，它们之间的接口应该是什么样的？

这是层次化嵌套架构中最关键的工程问题。本节给出答案——**卦象意图通讯协议**（Hexagram Intent Communication Protocol, HICP）。

---

## 7.3.1 传统跨层通讯方式的缺陷

在回答"卦象意图是什么"之前，先分析一下**为什么不能使用传统的跨层通讯方式**。

**数值指令模式**（传统方案）：高层向低层发送目标关节角度、目标末端位姿等数值指令。问题是：低层不理解"为什么是这个目标"，高层不理解"为什么低层执行得不好"。层之间的"理解"是完全脱节的。一个关节在追踪目标角度时遇到了力矩过载——低层只能报告"跟踪误差过大"，但它没法说"因为这个速度太快了我承受不了"；高层也听不到这种解释。

**自然语言指令模式**（LLM方案）：高层向低层发送自然语言指令——"左腿向前迈步30厘米"。这种模式的问题是：自然语言缺乏精确的语义层次，一个句子可以包含情境描述、动作方向和力度建议的混合信息——低层在解构混合信息时可能出错。而且自然语言的传输带宽远高于实际需要的语义带宽（"左腿向前迈步30厘米"至少需要8字节的UTF-8编码，而真正需要传递的语义只有"步态模式=行走、方向=前、幅度=0.3m"三个信息，编码为3个标量足矣）。

**混合模式**（ROS/典型机器人框架）：高层用A*规划出全局路径（一系列离散路点），低层用MPC追踪路点。接口是固定格式的导航消息——路点列表、期望速度、容忍度。问题是：这种接口绑定了高层的规划算法类型（基于路径规划）和低层的控制算法类型（基于模型预测控制）。如果高层换用RRT*规划器，消息格式不变（仍然是路点列表），但RRT*的路点稀疏度高，低层MPC在稀疏路点之间的插值行为可能不符合预期。

这三种传统方案的共同缺陷是：**层间通讯绑定了算法类型，限制了各层的独立演化。** YLYW的卦象意图协议的设计目标就是解除这种绑定——让高层和低层之间传递的不是算法参数，而是共享语义。

---

## 7.3.2 卦象意图消息格式

卦象意图是一个结构化的消息帧，由四个字段组成。

**卦象意图消息格式：**
```
{
  "hexagram": <卦名/卦序>,      // 必选。核心语义
  "trigram_sequence": <6爻序列>, // 必选。六爻值向量
  "urgency": <0-5>,            // 可选。紧急程度，默认0
  "constraint": <约束数组>      // 可选。额外的约束条件
}
```

- **hexagram字段**（6bit）：卦名或卦序。这是消息的核心语义——发送层告诉接收层"我认为当前状态属于X卦"。接收层的L3匹配引擎将这一卦象作为先验偏置，指导自己的推理过程，但不是强制命令。
- **trigram_sequence字段**（6位浮点向量，每个值∈[0,1]）：六爻值向量，给出了发送层对当前六维状态的"最新印象"。接收层可以将这些爻值作为自己L2编码的输入的一部分，但也可以覆盖——如果自己的传感器数据与接收到的六爻有冲突，以自己的传感器为准。
- **urgency字段**（0-5标量）：紧急程度。0=常规（低层可自主决定是否采纳），3=建议（低层应优先考虑），5=强制（低层必须执行，不可降级或抵抗）。紧急程度5的卦象意图通常只在安全八卦发出CRITICAL预警时使用。
- **constraint字段**（可选数组）：额外的约束条件列表，如["最大速度不超过0.5m/s", "保持力矩在额定值的80%以内"]。这类约束是对卦象意图的"精调"——高层知道宏观的任务约束（"杯子里有热水，所以速度不要太快"），但不想改变卦象类型（仍然是"行走"而非"静止"），因此以约束数组的形式附加在卦象意图消息中。

**消息体本身的物理大小仅为几个字节**——6bit的卦象编码 + 6位浮点值 + 一个标量 + 可选的少量约束参数。与传统的路点列表（数百浮点数）或自然语言指令（数百字符）相比，卦象意图的带宽降低了2-3个数量级。

## 7.3.3 卦象意图的三个核心特性

卦象意图与传统跨层消息的根本区别在于三个特性：**抽象性、可计算性、天生安全性**。

**抽象性。** 卦象意图不包含任何数值执行参数。它发送的不是"左腿向前迈30厘米、速度0.5m/s、加速度1.0m/s²"这种完整的运动学指令——它发送的是"当前状态为火风鼎䷱——稳步前进时应保持当前速度与姿态"。接收层自行解读"稳步前进"意味着什么——对自己的子系统应该输出什么参数。

这种抽象性的工程优势是：高层不需要知道低层的执行能力——它不知道低层是关节空间控制还是任务空间控制、不知道低层的力矩上限值、不知道低层的控制周期。它唯一需要知道的是：低层理解"火风鼎"的卦象语义。如果低层是一个PID关节控制器的YLYW元胞，它从"火风鼎"中解读为"保持当前关节角度，微调P增益以减小跟踪误差"；如果低层是一个阻抗控制器的YLYW元胞，它从"火风鼎"中解读为"保持当前末端位姿，降低刚度以提高柔顺性"。同一条卦象意图，不同的低层实现有不同的解读。高层不需要为每种低层实现写不同的消息。

**可计算性。** 卦象意图可以使用第3.4节的爻位关系算子进行**跨层推理**。当微观层报告的爻位序列与宏观层本地感知到的爻位序列不一致时——例如微观层的六爻显示"阴阴阴阴阴阳"（推搡后的不稳定状态），而宏观层的六爻显示"阳阳阳阳阳阳"（全局状态正常，因为宏观层不直接感知单个关节的力矩变化）——宏观层可以使用"呼应"算子分析这两条六爻序列之间的差距："初爻同性格（阳-阴不同），四爻同性格（阳-阴不同）……呼应数为0，表示宏观与微观之间存在系统性的状态感知偏差。"这个跨层推理的结果是一个卦象——"䷿ 火水未济——宏观与微观之间各爻的匹配度极低。应采取校正措施。"这就是"可计算性"的含义：**卦象不仅仅是描述性的标签——它是推理的输入和输出。**

**天生安全性。** 当高层发送"奔跑！—— ䷡ 雷天大壮"的卦象意图时，低层不是无条件执行。低层用自己的安全八卦独立检查这个卦象在当前物理条件下是否安全。如果安全八卦检测到三爻（关节力矩）或四爻（关节温度）处于潜在危险区域，它会**拒绝执行**——将卦象降级为"䷴ 风山渐（慢跑——低力矩模式）"——而不是无脑追随高层的指令。这种"每层都安全检查"的机制，确保了即使是高层发出了一个"错误"的卦象意图（可能来自指令错误或感知误解），低层也会在物理层面拦截它。

## 7.3.4 上行反馈：变卦预警

卦象意图协议不仅支持从上到下的指令下传，还支持从下到上的状态上传。但这个"上传"不是传统意义上的数据回传——它是**变卦预警**。

当一个YLYW元胞（无论是微观层还是中观层）检测到自己的六爻序列正在向显著不同的卦象漂移时——例如当前最佳匹配卦象的相似度从0.95降至0.70，而备选卦象"坎为水䷜"的相似度从0.40升至0.70——它触发一个变卦预警，向自己的上层报告。

**变卦预警消息格式：**
```
{
  "from_hexagram": <当前卦象>,
  "to_hexagram": <目标卦象>,      // 正在漂移到的卦象
  "trigger_yao": <触发爻位>,       // 哪条爻的变化触发了预警
  "values": <当前六爻值向量>,
  "trend": <变化趋势描述>          // 如"三爻1.5s内从0.65升至0.88(老阳逼近)"
}
```

**案例分析：膝关节力矩接近极限的过程。** 这是一个完整的"三角形"卦象预警流程：

**Step 1（微观层→中观层）：** 膝关节YLYW检测到三爻(力矩)在1.5秒内从0.65升至0.88。触发变卦预警——"当前卦䷌（天火同人—正常大步行走），三爻逼近老阳(0.92)，建议切换至䷨（山泽损—减力策略）。"

**Step 2（中观层内部）：** 双腿元胞YLYW收到变异预警。它自己的爻位没有剧烈变化（自身的力矩爻位还在安全范围），但考虑到宏观层指令是"大步行走"，而微观层已经报告力矩逼近极限——双腿元胞使用"乘承"算子分析：宏观层的指令（大步行走）在微观层（力矩负荷）上产生"阴乘阳"效应（高指令压低了低层的安全裕度）。因此双腿元胞执行卦象降级：从"䷡ 雷天大壮（大步行走）"→"䷴ 风山渐（小步慢走）"。

**Step 3（中观层→宏观层）：** 双腿元胞向宏观层发送一个普通预警（urgency=2）——"䷡→䷴ 因膝关节力矩逼近上限，步态从大步降级为小步。当前力矩值0.88，安全阈值0.90。"宏观层收到预警后更新自己的五爻（安全度：从阳降至阴），但认为其他爻位正常，保持"䷊ 泰卦"全局策略。

**最终效果：** 力矩没有超过阈值（在0.88处停止增长并回落），关节没有损坏，任务没有中断。三个层级在统一符号框架下，自主完成了从"检测风险→预警→降级→报告"的安全协同。没有任何数值要守——整个过程发生在卦象语义空间中。

## 7.3.5 与蚁群智能的系统类比

卦象意图通讯协议在机制上与蚁群智能之间具有显著的结构相似性。

在蚁群中，单个蚂蚁通过释放**信息素**与群体沟通。信息素不是直接的数值指令（"向左转30度"），而是间接的**状态标识**（"这条路径通往食物源"）。其他蚂蚁通过感知信息素的浓度来调整自己的行为——不是被动的指令执行，而是主动的"情境感知"。

在YLYW的嵌套架构中，卦象意图起到了类似信息素的作用。宏观层广播的卦象意图不是"你必须执行X动作"的强制指令——它是"当前全局状态为X卦象"的状态广播。中观层收到这个广播后，根据自己当前的子系统和环境条件，自主决定如何响应这个卦象。中观层与中观层之间不需要直接数据传输——它们通过对同一卦象意图的独立解读自然协调（双腿元胞解读"大壮"为快步前进，双臂元胞解读同一个"大壮"为快速摆臂）。

**表7.4 蚁群智能与YLYW卦象意图对比**

| 蚁群机制 | YLYW对应 | 功能 |
|---------|---------|------|
| 信息素标记 | 卦象意图广播 | 向群体传递环境状态信息 |
| 个体独立解读信息素 | 子系统独立解码卦象 | 个体根据自身状态调整行为 |
| 间接协调 | 零显式子系统通讯 | 全局协调通过共享状态而非直接协商 |
| 信息素浓度→行为强度 | 卦象紧急度→执行优先级 | 紧急状态对应高优响应 |
| 无中心协调者 | 宏观层是发送者，不是控制器 | 不存在单点故障 |

"无中心协调者"是蚁群智能最关键的性质——蚁群不依靠一个"指挥蚁"来调度一切工作。YLYW的嵌套架构同样没有中心控制器。宏观层确实是卦象意图的广播者，但它不承担"规划一切任务"的角色——中观层和微观层对卦象意图的解读和执行是自主的。如果宏观层损坏（例如电机关联系统故障），中观层和微观层仍然可以基于当前系统状态继续工作一段时间——它们不依赖单一控制节点。
